- **validated: kitty `hide_window_decorations yes` was hardcoded in
  `render_kitty_conf` (setup.rs) — broke window chrome on stacking WMs.**
  Reproduced live on this machine: KDE Plasma/kwin/Wayland
  (`XDG_CURRENT_DESKTOP=KDE`), user lost title bar + min/max/close + resize
  border entirely after `8sync setup --profile terminal`. Root cause: the
  setting is meant for tiling compositors (Hyprland/HyDE) that draw no chrome
  themselves; on kwin/mutter (stacking), the compositor doesn't add
  server-side decorations for an undecorated client either — so hiding
  kitty's own decorations leaves NOTHING. Fix: `env_detect::is_tiling_wm()`
  (checks `is_hyde()` + `XDG_CURRENT_DESKTOP`/`DESKTOP_SESSION` against
  hyprland/sway/river/wayfire/qtile/i3/bspwm/awesome) gates the setting in
  `render_kitty_conf`. Verified: re-ran `8sync setup --profile terminal` with
  the fixed binary, `~/.config/kitty/8sync.conf` no longer contains the line.
  **Caveat**: decorations are negotiated at window-creation (Wayland
  xdg-decoration protocol) — `kitty @ load-config` does NOT retroactively
  restore them on an already-open window; user must close/reopen the kitty
  window.
- **validated: `zai-vision` MCP (`@z_ai/mcp-server`) bridges GLM-5.2's text-only
  gap — tested end-to-end via real stdio JSON-RPC `tools/call`, not just direct
  API.** Package tools take `image_source` as a **file path or URL** (not
  base64) + `prompt` — trivial to call from any context. **failure→fix: the
  server always sends `thinking:{type:'enabled'}` and a hardcoded model
  default; on a stock `zai/glm-5.2` Coding-Plan key every PAID vision model
  (`glm-4.6v`, `glm-4.5v`, `glm-5v-turbo`, `glm-ocr`) 400s with `1113
  insufficient balance` regardless of `thinking`.** Only **`glm-4.6v-flash`**
  (confirmed FREE on Z.AI's live pricing page) works on a stock key — verified
  3× through the actual MCP server, both with and without `thinking`.
  `1301`/`1305` looked at first like `thinking`-related bugs but retesting
  proved they're just transient noise (same model, same body, flips between
  success/1305 across attempts) — don't over-fit a single failed call to a
  root cause; retry 2-3× before concluding. Fix: `8sync harness` now defaults
  `Z_AI_VISION_MODEL=glm-4.6v-flash` when registering the MCP
  (`deploy.rs::ensure_zai_vision_mcp`). Key resolution: `omp token zai` (omp's
  auth-broker) returns the plain key on stdout — same key auths `zai/glm-5.2`.
  omp's `stdioServer` mcp.json schema supports `env` (confirmed via the
  upstream schema URL) — `register_omp_mcp` extended to emit it only when
  non-empty so existing env-less servers stay byte-identical (no spurious
  self-heal rewrites).
